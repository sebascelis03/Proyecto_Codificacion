# Workshop: Spec-Driven Development (SDD) with Kiro AWS

## Course: Advanced Network Application Design

### Objective

In this workshop, you will learn how to use **Kiro AWS** to apply **Spec-Driven Development (SDD)** — a methodology where you define clear specifications *before* writing any code. Kiro uses three spec files (`requirements.md`, `design.md`, `tasks.md`) to guide the entire development process.

You will build a **Point of Sale (POS) Frontend** for a supermarket, starting from specifications only.

---

## What is Spec-Driven Development?

SDD is a development approach where:

1. You define **what** the system should do (requirements)
2. You define **how** the system should be built (design)
3. You break the work into **actionable tasks** (tasks)
4. The AI agent (Kiro) implements the code based on these specs

This ensures consistency, traceability, and clarity throughout the development lifecycle.

## Workshop Steps

### Step 1: Create the Project Spec Folder

In your Kiro workspace, create a `.kiro/specs/pos-frontend/` folder. This is where the three spec files will live.

```
.kiro/
  specs/
    pos-frontend/
      requirements.md
      design.md
      tasks.md
```

### Step 2: Understand the Spec Files

| File | Purpose |
|------|---------|
| `requirements.md` | Defines WHAT the system must do. User stories, acceptance criteria, functional requirements. |
| `design.md` | Defines HOW the system will be built. Architecture, components, data flow, tech stack. |
| `tasks.md` | Defines the WORK to be done. Ordered implementation tasks with checkboxes for Kiro to execute. |

### Step 3: Review the Provided Specs

The three spec files have been pre-written for this workshop. Open each one and review:

1. **`requirements.md`** — Read every user story and its acceptance criteria carefully
2. **`design.md`** — Understand the component architecture and data flow
3. **`tasks.md`** — See how the work is broken into incremental, testable steps

### Step 4: Let Kiro Implement

Once you understand the specs, open the Kiro Spec panel and select the `pos-frontend` spec. Kiro will:

1. Read the requirements to understand the scope
2. Follow the design to make architectural decisions
3. Execute tasks one by one, checking them off as completed

### Step 5: Validate and Iterate

After Kiro completes each task:

- Review the generated code
- Run the application locally
- Verify it meets the acceptance criteria from `requirements.md`
- If something is wrong, update the spec and let Kiro re-implement

## Project Context: Supermarket POS Frontend

You are building a **Point of Sale (POS) terminal interface** for a supermarket cashier. The system must:

- Allow cashiers to search products by name or category
- Support barcode scanning (manual entry + camera-based scanning)
- Display a real-time shopping cart with running totals
- Apply discounts and calculate taxes
- Process checkout with multiple payment methods
- Generate and display a digital receipt
- Work offline for core operations (product search, cart management)

## Tips

- Be as specific as possible in your specs — vague requirements produce vague code
- Use acceptance criteria with concrete, testable conditions
- Keep tasks small and incremental — Kiro works best with focused tasks
- If Kiro generates something unexpected, check your spec first before blaming the AI
